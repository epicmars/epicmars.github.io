LDAP服务与认证

公司团队需要搭建项目管理、代码托管等服务器，目前使用redmine，gerrit，gitlab，jenkins等等，对于每个服务器都需要为每个成员建立账户，重复的工作太多，每次让成员在首次登录时重新设置密码也不太人性化。正好这些服务都支持LDAP认证，那么建立一个公用的认证服务相当有必要了。事实上公司内部账户系统通常采用这种方式搭建。

什么是目录服务(Directory service)

一个目录是一个特殊的数据库，设计用于搜索和浏览，也支持基本的查询和更新功能。

注意：目录服务某种程度上仅仅被定义为对读取访问进行了优化的数据库，这一定义是过于简化了的。

目录趋向于包含描述性的和基于属性的信息，并且支持复杂的过滤能力。目录通常不支持数据库管理系统中设计用来处理大容量复杂更新的复杂事务或者回退机制。如果允许的话，目录的更新一般仅仅是简单地进行全量更新或者什么也不改变。通常目录经过调优后，可以对大容量查询和搜索快速做出响应。为了增加可用性和稳定性，同时降低响应时间，它们可能具有对信息进行广泛复制(replicate)的能力。当目录信息被复制后，复本间的临时性的不一致或许还说的过去，只要其不一致可以在时间顺序上进行解决。

什么是LDAP

LDAP是Lightweight Directory Access Protocol的缩写，即轻量级目录访问协议。它是一个用于访问目录服务的协议，尤其是基于X.500的目录服务。LDAP运行于TCP/IP上，或其他面向连接的传输服务。

RFC4510对其技术规范路线图进行了概述。

这一节从用户的视角对LDAP进行了概述。

什么样的信息可以存储在目录中？LDAP信息模型基于条目(entries)。一个条目是一个属性集合，它拥有一个全局唯一的专有名称(DN, Distinguished Name)。DN用于明确地指向一个条目。每个条目的属性有一个类型和一个或多个值。类型常常是助记字符串，例如“cn”表示common name，或者“mail”表示email地址。值的语法取决于属性类型。例如一个cn属性可以包含值Babs Jensen。一个email属性可以包含值"babs@example.com”。一个jpge图像属性可以包含一个JPEG（二进制）格式的图像。

信息是如何进行组织的？在LDAP中，目录条目组织成一个树形结构。通常，这一结构反应了地理和(或)组织的边界。代表国家的条目处于树的最顶端，以下是一个代表州和国家机构的条目，他们下面的条目代表组织单位，人员，打印机，文档或者其它你可以想象到的其它任何东西。图一展示了一个使用传统命名的LDAP树。

如何引用信息？

一个条目由其专有名称(DN, Distinguished Name)进行引用，它是通过将条目名称自身(称为相对专有名称，Relative Distinguished Name 或者 RDN)和其祖先条目名称连接起来的。例如，在因特网命名示例中Barbara Jensen有一个RDN为uid=babs以及一个DN为uid=babs,ou=People,dc=example,dc=com。完整的DN格式在RFC4514的LDAP: String Representation of Distinguished Names章节进行了描述。

如何访问信息？

LDAP定义了用于查询和更新目录的操作。提供的操作用于从目录中添加或删除一个条目，改变一个已有的条目，以及改变一个条目的名称。然而，大多数时候，LDAP用于在目录中搜索信息。对于与某一搜索过滤器所指定的某些标准相匹配的条目，LDAP搜索操作允许搜索目录某些部分。信息可以从匹配搜索标准的每个条目中获取。

例如你可能想搜索位于dc=example,dc=com及其下面整个目录子树中名为Barbara Jensen的people，检索发现的每个条目的email地址。LDAP可以让你轻易做到。或者你想直接搜索位于条目 st=California,c=US下名为Acme的并且有一个fax number的origanization。LDAP也能让你做到。

如何保护信息不受未授权的访问？

有些目录没有提供任何保护措施，允许任何人进行访问。LDAP为客户端提供了一种认证机制，或者说让客户端来提供访问目录服务的证明，这样为丰富的访问控制以及对信息的保护提供了一条方便的道路。LDAP也支持数据安全服务。

什么时候应当使用LDAP？

非常好的问题，通常当你需要采用集中式的方式管理数据，并通过基于标准的方法存储与访问数据时你应当使用目录服务。

工业上常见的例子包括但不限与如下所列出的这些：

  • 机器认证（Machine Authentication）
  • 用户认证（User Authentication）
  • 用户/系统组（User/System Groups）
  • 地址簿（Address book）
  • 组织表示（Organization Representation）
  • 资产追踪（Asset Tracking）
  • 电话信息存储（Telephony Information Store）
  • 用户资源管理（User resource management）
  • Email地址查询（E-mail address lookups）
  • 应用配置存储（Application Configuration store）
  • PBX配置存储（PBX Configuration store）
  • 等等…

有各种基于标准的分布式schema文件(Distributed Schema Files )，但你总是可以创建你自己的Schema规范。

总是有新的方式来使用一个目录并应用LDAP原则来处理特定的问题，因此没有简单的答案来回答这个问题。

关于X.500

技术上而言，LDAP是一个对X.500目录服务进行访问的目录访问协议，即OSI目录服务。从LDAP客户端访问到达X.500目录服务的网关开始，该网关在客户端和网关间运行LDAP并且在X.500服务和网关间运行目录访问协议(DAP, Directory Access Protocol)，DAP是一个重量级的协议，它可以在一整个OSI协议栈上进行操作，并且需要相当多的计算机资源。LDAP设计为在TCP/IP上进行操作并且以更低的消耗来提供DAP的大多数功能。

尽管LDAP仍然可以通过网关访问X.500目录服务，LDAP现在更常见的是直接实现在X.500服务器中。

单一的LDAP守护进程或者slapd可以被看做一个轻量级的X.500目录服务器。也就是说它不实现X.500 DAP也不支持完整的X.500模式。

LDAP vs RDBMS

为什么OpenLDAP不使用一个关系型数据库管理系统(RDBMS)而是使用一个像LMDB这样的嵌入式的键/值存储系统？这样我们可以期待商业级RDBMS实现的复杂算法可以使OpenLDAP更快或者更好，同时允许与其它应用进行数据分享。

简单的回答就是使用嵌入式数据库和定制的索引系统使得OpenLDAP可以提供您更高的性能，并且在不损失可靠性的情况下拥有更强的可扩展性。OpenLDAP使用LMDB并发的/事务性的数据库软件。

详细的答案就是：我们总是面临着在RDBMS和目录数据库间做出选择，这是一个难题并不存在简单的答案。

可能会想到如果有一个由RDBMS支撑的目录服务那么所有问题就迎刃而解了。然而事实并非如此，因为他们的数据模型是相当不同的。使用一个关系型数据库来表示目录数据需要将数据划分到不同的表中。

考虑person objectclass的例子。其定义中需要属性类型objectclass,sn和cn以及属性类型userPassword,telephoneNumber,seeAlso和描述。所有这些属性都是具有多个值的，因此对数据做一个规范化需要将每个属性类型放到一个单独的表中。

现在你必须为这些表确定合适的键。主键可能是DN的组合，但这在大多数数据库实现中都相当低效。

现在更大的问题是从一个条目访问数据需要去搜索不同的磁盘区域。在一些应用中还OK，但在其它许多应用中会存在性能问题。

唯一可以放在主表条目中的属性类型是哪些强制要求并且具有单个值的。你也可以将可选的单个值的属性添加到其中，并且如果不存在的话就将它们设置为NULL。

但是等一下，条目可以具有多个objectclass，并且它们以继承层次结构进行组织。一个organizationalPerson的objectclass条目现在有从person继承来的属性，加上一些其它条目继承来的属性以及一些之前可选的属性类型，现在都变成了强制性要求。

那么该怎么做？为每个不同的objectclass建立不同的表？这样一个person在person表中有一个条目，在organizationalPerson中也有一个条目，等等。或者我们是否应该去掉person表，将所有数据都放到第二个表中。

但如果有一个类似(cn=*)的过滤器，其中cn是一个在很多、很多objectclass中出现过的属性类型。我们是否应该搜索所有可能的表格来寻找匹配的条目？这样做并不讨喜。

LDAP服务器

OpenLDAP

OpenLDAP是对LDAP的开源实现，包括一个独立的LDAP服务器slapd，实现LDAP协议的库，以及相关组件、工具和简单的客户端。

Apache Directory

Apache Directory是Apache的一个项目，包含多个目录服务器相关的子项目，其中最核心的就是使用Java编写的可扩展以及嵌入式目录服务器ApacheDS。

Active Directory

简称AD（中国大陆译名为“活动目录”，台湾为维持英文不译），微软Windows Server中，负责架构中大型网络环境的集中式目录管理服务。
