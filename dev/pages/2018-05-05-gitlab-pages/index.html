Gitlab Pages功能在默认情况下是未开启的，首先需要先开启该功能，首先修改Gitlab配置文件：

sudo vi /etc/gitlab/gitlab.rb

修改如下一行为：

gitlab_pages['enable'] = true

然后退出vi并重新配置，使修改的配置生效，并重启Gitlab Pages，每次修改配置文件都需要进行以下操作：

sudo gitlab-clt reconfigure

sudo gitlab-ctl restart gitlab-pages
# 或者
sudo gitlab-ctl restart

为了搭建一个Gitlab Pages站点，需要完成如下步骤：

  1. 一个工程
  2. 一个用于部署站点的配置文件(.gitlab-ci.yml)
  3. 配置文件中的一个称为pages的指定job，可以让Gitlab识别出你在部署一个Gitlab Pages站点
  4. 一个public文件夹，里面是站点内容

可选的特性包括：

  1. 一个自定义域名或子域名
  2. 一个指向你的站点(子)域名的DNS，此外还可以为自定义域名配置SSL/TLS证书来使用HTTPS进行访问

首先在Gitlab上创建一个空的项目，地址为git@yourdomain.com:pages/jekyll.git，然后克隆官方的jekyll示例项目：

git clone https://gitlab.com/pages/jekyll.git

或者在本地使用jekyll命令行工具生成一个初始项目：

jekyll new jekyll

然后执行如下命令：

cd jekyll
git remote rename origin old-origin
git remote add origin git@yourdomain.com:pages/jekyll.git
git push -u origin --all
git push -u origin --tags

在Gitlab上查看，工程成功上传了，并且查看项目下的CI/CD>Jobs可以看到一个任务被触发了，状态为Pending，点击状态进入任务的详情页面显示如下信息：

This job is stuck, because the project doesn't have any runners online assigned to it.

进入工程下的Setting>CI/CD，查看是否存在运行中的Runner，发现还没有配置Runner。

Gitlab Runner

Gitlab Runner是Gitlab持续集成(Gitlab CI)的执行器，它执行定义在.gitlab-ci.yml中的命令或任务。它们是独立的（虚拟）机器，通过Gitlab CI的协作API来获取任务。

一个Runner可以是指定用于一个特定的项目，也可以服务于Gitlab CI中的任意项目。一个服务所有项目的Runner称为共享Runner。

理想情况下Gitlab Runner不应当安装到Gitlab所在的同一台服务器上。

共享与特定的Runner

在安装Gitlab Runner后，可以将其注册为共享或特殊的。

  • 共享Runner：对于多个项目间有相同需求的任务十分有用。

  • 特殊Runner：对于有特殊需求的项目或有特殊需求的任务十分有用。

安装Gitlab Runner

首先进行Gitlab Runner的安装，因为Gitlab Runner会消耗大量内存，以及可能的安全隐患，尤其是以shell执行器的方式使用Gitlab Runner。如果有多台服务器可供使用，官方推荐的做法是将Runner安装到与Gitlab不同的另外的服务器上：

Gitlab Runner的注册

所谓注册，就是将一个Runner绑定到一个Gitlab实例，然后它们之间就可以进行协作了，因为Runner是另外的进程，也有可能安装在与Gitlab不同的服务器上。

注册共享Runner

注：只有管理员才可以注册一个共享Runner。

  1. 在admin/runners页面获取共享Runner令牌(Token)

  2. 注册Runner

    在注册Runner前，需要完成Gitlab Runner的安装，并获取一个令牌。

    在没有特殊需求的情况下，Runner Executor选择shell即可。注册完成后刷新admin/runners页面，可以发现注册的Runner的信息出现在了下方的列表中。

Gitlab Pages搭建问题与解决方法

jekyll站点首页未加载css以及访问二级页面返回404的问题

最初在初始化站点项目时，采用的是命令行初始化的方式，jekyll配置文件_config.yml未配置url路径，与Gitlab上的jekyll站点示例项目对比发现，进行如下配置可以解决问题：

baseurl: "/jekyll" # the subpath of your site, e.g. /blog
url: "/" # the base hostname & protocol for your site, e.g. http://example.com
