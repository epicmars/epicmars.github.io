android.net.wifi

Wi-Fi API为应用提供了一种方法，使其可以和提供Wi-Fi网络访问的底层无线技术栈进行通信。所提供的API几乎可以获取设备所有信息，包括连接网络的速度，IP地址，协商状态等，以及关于其它网络的信息。一些其它API特性包括扫描、添加、保存、终止和初始化Wi-Fi连接。

一些API需要以下用户权限：

  • ACCESS_WIFI_STATE

    允许应用访问关于Wi-Fi网络的信息

  • CHANGE_WIFI_STATE

    允许应用改变Wi-Fi连接状态

  • CHANGE_WIFI_MULTICAST_STATE

    允许应用进入Wi-Fi多播(Multicast)模式

注意：并非所有的Android设备都提供了Wi-Fi功能，如果应用使用Wi-Fi才能运行，那么应当在manifest文件中使用一个<uses-feature>元素进行配置。

    <manifest ...>
        <uses-feature android:name="android.hardware.wifi" />
        ...
    </manifest>

核心类

  • ScanResult

    描述一个检测到的AP的信息，除了此处描述的属性外，客户端还跟踪其质量、噪音、最大波特率特性，但不将它们报告给外部客户端。

    主要属性包括：

    • BSSID

      AP的地址

    • SSID

      网络名称

    • capabilities

      描述AP所支持的认证，秘钥管理与加密机制

    • level

      以dBm为单位的信号等级，也称为RSSI

  • WifiConfiguration

    表示一个配置过的网络，包括安全配置。

  • WifiEnterpriseConfig

    Wi-Fi企业配置详情。

  • WifiInfo

    描述任意活跃的或正在被建立的Wifi连接的状态。

  • WifiManager

    提供了管理Wi-Fi连接的所有方面的主要API。

    它处理若干类别的项目：

    • 配置过的网络列表，列表可以查看并更新，并且可以修改每个独立入口的属性
    • 当前活跃的Wi-Fi网络
    • AP扫描的结果
    • 它定义了在任意类型的Wi-Fi状态改变时所广播出去的各种类型Intent操作的名称

    在进行特定Wi-Fi操作时需要使用这些API，要在一个抽象层级上对网络连接进行操作，应当使用ConnectivityManager。

  • WifiManager.WifiLock

    允许应用保持Wi-Fi无线电处于开启状态。通常当用户有一会没有使用设备时，Wi-Fi无线电可能会关闭。获取一个WifiLock会保持无线电处于开启状态直到锁被释放。多个应用可以持有WifiLock，只有当没有应用持有琐时，无线电才允许关闭。

    在使用WifiLock前应当考虑清楚，你的应用是否需要Wi-Fi访问，或者，如果有的话可以在移动网络上正常工作。一个需要下载大型文件的应用应该持有一个锁保证下载完成，但一个网络使用只是临时的或者低带宽的不应当持有一个WifiLock以避免对电池寿命产生不利影响。

    注意WifiLock不能覆盖用户级的“Wi-Fi 开启”设置，以及飞行模式，他们仅仅是在Wi-Fi已经开启但设备处于空闲时防止无线电关闭而已。

    任何使用一个WifiLock的应用必须在清单文件中加入<uses-permission android:name="android.permission.WAKE_LOCK"/>元素。

android.net.wifi.aware

提供了使用Wi-Fi Aware来发现对等设备并进行连接的类。

使用Wi-Fi Aware API，应用可以发布服务，发现正在发布服务的的对等设备并进行连接。Wi-Fi Aware是独立于Wi-Fi基础设施的（也就是一个设备在使用Wi-Fi Aware的同时可能或者没有关联到一个AP）。

Wi-Fi的主要入口是WifiAwareManager，通过调用Context.getSystemService(Context.WIFI_AWARE_SERVICE)来获取。

一些API需要如下用户权限：

  • ACCESS_WIFI_STATE

    允许应用访问关于Wi-Fi网络的信息

  • CHANGE_WIFI_STATE

    允许应用改变Wi-Fi连接状态

  • ACCESS_COARSE_LOCATION

    允许应用访问近似位置。可选的，你可能想要ACCESS_FINE_LOCATION

注意：并非所有的设备都支持Wi-Fi Aware功能，如果应用必须使用Wi-Fi Aware才能运行，那么可以在配置文件中声名一个<uses-feature>元素。

    <manifest ...>
        <uses-feature android:name="android.hardware.wifi.aware" />
        ...
    </manifest>

可选地，如果你的应用不需要Wi-Fi Aware，但如果设备支持的话可以加以利用，可以在运行时使用hasSystemFeature(String)和FEATURE_WIFI_AWARE进行检查。

    getPackageManager().hasSystemFeature(PackageManager.FEATURE_WIFI_AWARE)

android.net.wifi.hotspot2.omadm

  • PpsMoParser

    一个工具类，用于将OMA-DM (Open Mobile Alliance’s Device Management) PPS-MO (PerProviderSubscription Management Object) XML树转换为一个PasspointConfiguration对象。

android.net.wifi.hotspot2.pps

  • Credential

    代表PerProviderSubscription (PPS) Management Object (MO) 树中的凭证子树。

  • Credential.CertificateCredential

    基于证书的凭证(Certificate based credential)。

  • Credential.SimCredential

    基于SIM (Subscriber Identify Module)的凭证。

  • Credential.UserCredential

    基于用户名-密码的凭证。

  • HomeSp

    代表PerProviderSubscription (PPS) Management Object (MO)树中的HomeSP子树。

android.net.wifi.hotspot2

类

  • ConfigParser

    从一个安装文件构建PasspointConfiguration的工具类。

  • PasspointConfiguration

    Passpoint配置。这包含了PerProviderSubscription (PPS) Management Object (MO) 树中指定的配置。更多信息可以参考Hotspot 2.0 Release 2 技术规范9.1节中定义的Hotpot 2.0 PPS MO。

android.net.wifi.p2p

提供了使用Wi-Fi Direct创建点对点(P2P)连接的类。

使用这些类，你可以发现支持Wi-Fi Direct的设备并与其建立连接，然后可以在一个快速的连接上进行通信，其距离比蓝牙连接更长。主要使用的类是WifiP2pManager，可以通过调用getSystemService(WIFI_P2P_SERVICE)来获取。

WifiP2pManager提供的API允许你进行如下操作：

  • 调用initialize()来为P2P连接初始化你的应用。

  • 调用discoverPeers()来发现附近的设备。

  • 调用connect()来发起一个P2P连接。

  • 更多其它操作

若干其它接口和类也是必要的，例如：

  • WifiP2pManager.ActionListener接口，当一个操作，例如发现对等设备或者连接到它们时，无论失败还是成功，允许你接收该事件回调。

  • WifiP2pManager.PeerListListener接口，允许你接收所发现的对等设备的信息。该回调提供了一个WifiP2pDeviceList，从中可以为每个在发现范围内的设备获取一个WifiP2pDevice对象，并获取例如设备名称，地址，设备类型，设备支持的WPS配置等信息。

  • WifiP2pManager.GroupInfoListener接口，允许你接收关于一个P2P组的信息。该回调提供了一个WifiP2pGroup对象，它提供的组信息诸如所有则，网络名称以及通行码(passphrase)。

  • WifiP2pManager.ConnectionInfoListener接口，允许你接收关于当前连接的信息。该回调提供了一个WifiP2pInfo对象，它拥有诸如一个组是否已经构成以及谁是组的所有者。

为了使用Wi-Fi P2P API，你的app必须遵循以下用户权限：

  • ACCESS_WIFI_STATE

    允许应用访问关于Wi-Fi网络的信息

  • CHANGE_WIFI_STATE

    允许应用改变Wi-Fi连接状态

  • INTERNET(尽管你的app从技术上讲没有连接到英特网，但使用标准java socket与Wi-Fi Direct对等设备进行通信需要英特网权限)

    允许应用打开网络socket。

同样的，并不是所有的设备都支持Wi-Fi Direct，如果你的应用使用Wi-Fi Direct，可以在配置文件中定义一个<uses-feature>元素。

    <manifest ...>
        <uses-feature android:name="android.hardware.wifi.direct" />
        ...
    </manifest>

android.net.wifi.rtt

Wi-Fi RTT(IEEE 802.11)可以让应用测量到支持Wi-Fi RTT的AP和其它对等设备的距离。

Wi-Fi RTT的主要入口是WifiRttManager类，调用Context.getSystemService(Context.WIFI_RTT_RANGING_SERVICE)来获取。

一些API需要如下用户权限：

  • ACCESS_WIFI_STATE

    允许引用访问关于Wi-Fi网络的信息

  • CHANGE_WIFI_STATE

    允许应用改变Wi-Fi连接状态

  • ACCESS_COARSE_LOCATION

    允许应用访问近似位置。可选的，你可能想要ACCESS_FINE_LOCATION

该API的使用受设备的位置模式控制：即是否允许查询基于Wi-Fi的位置。

注意：并非所有的设备都支持Wi-Fi RTT功能，如果应用必须使用Wi-Fi RTT才能运行，那么可以在配置文件中声名一个<uses-feature>元素。

    <manifest ...>
        <uses-feature android:name="android.hardware.wifi.rtt" />
        ...
    </manifest>

可选地，如果你的应用不需要Wi-Fi RTT，但如果设备支持的话可以加以利用，可以在运行时使用hasSystemFeature(String)和FEATURE_WIFI_RTT进行检查。
