移动端跨平台技术比较

Cordova与Ionic

Cordova是使用Web技术（HTML/CSS/JavaScript）来构建移动应用的一种框架。它提供了一组标准的API来绑定设备所提供的功能，可以在不同平台上提供一致的开发体验。虽然它可以利用宿主平台的原生功能，但本质上它仍然是一个Web App。

可以看到Cordova是基于WebView的，这是其提供垮平台能力的关键之处，它是一个嵌入式的HTML渲染引擎，即浏览器引擎，为不同平台的应用提供一致的Web界面。同时它提供了一系列的插件，通过桥接方式，使得JavaScript代码可以访问系统提供的原生API。

Ionic构建于Cordova之上，提供了一个前端组件集合，使你能够编写接近原生界面的Web App。

  • 缺点：Web App是基于DOM的，DOM本身并不慢，但渲染（应用CSS样式、布局、绘制、合成）的过程很慢。如下图所示：

    DOM的操作是由JavaScript完成的，对DOM的改动会触会发布局重绘，由于JavaScript是单线程的，而且这部分工作不能利用GPU进行加速，因此Web界面交互和动画性能较弱。而原生App不存在这样的问题，例如Android中的主线程即UI线程主要用于视图绘制，其它耗时的任务则必须交给工作线程处理，以避免阻塞主线程，并且视图绘制可以利用GPU进行加速，因此Web App的用户体验难以与原生App相媲美。原生应用的其它优势还有，更丰富细腻的动画效果，更好的手势识别，有更适合的线程模型等等。

  • 优点：Cordova有着良好的跨平台特性，所谓“write once, run anywhere”。如果应用需要的功能主要是内容的展示，不存在太多的交互那么它无疑是一个较好的选择。

React Native

React Native是使用React来构建原生应用的一种框架。其理念不是“write once, run anywhere”，它并不是要一统三端（Web/iOS/Android），而是通过在不同平台间复用JavaScript代码，快速进行业务开发与迭代，即“learn once, write anywhere”。

React Native的内部实现是在APP中使用一个嵌入式的JavaScriptCore，即JavaScript虚拟机，作为JavaScript运行环境。然后通过JavaScript调用宿主平台的原生API，从而可以使用JavaScript开发出与原生应用用户体验相当的应用，而非运行于移动端的Web应用。因此在进行跨平台的开发时，可以利用JavaScript已有的API快速地进行业务迭代，对于JavaScript所没有的原生API，它通过一种桥接的方式，通过JavaScript来与原生API进行通信，充分利用了不同平台所具有的特性。它也并不打算实现一套自己的组件来完全替代这些原生的组件，因为这样做较为复杂，而且即使做了也未必能达到原生组件的性能。对于界面的渲染实际上采用的也是原生的UI组件，并不是一个Web页面，这是通过将JSX所定义的视图渲染成原生的视图组件达到的。

  • 缺点：

    • 尽管可以在不同平台间复用相当一部分JavaScript代码，但对特定平台的适配会导致复用比例下降。即需要利用平台特有的原生功能越多，平台间的JavaScript代码复用率越低。
    • 由于需要兼顾JavaScript和原生的API，其扩展性不及Web技术，也不及原生代码。
    • 相比与已经相当成熟的Web技术，React Native所采用的技术栈还不够成熟，目前版本还是0.55，许多功能还处于开发阶段。

  • 优点：

    • 采用了React框架的思想。
    • 媲美原生应用的性能。

总结

将Web App，React Native和原生App三种跨平台开发技术在技术成熟度、代码复用率，性能等方面做一个比较，因为Web技术相比其它技术的历史都更长，这里比较的范畴限定在移动开发的领域，按从高到低的顺序，有：

高 中 低
技术成熟度 原生App Web App React Native
代码复用率 Web App React Native 原生App
性能 原生App React Native Web App

实际开发中，以上技术方案并不是非此即彼的关系，在一款客户端中可以运用其中一种或同时采用多种技术进行开发。
