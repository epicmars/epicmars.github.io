Java内存模型：volatile变量

根据JLS 8.3.1.4，Java语言允许多个线程对共享的变量进行访问，作为一种规则，为了保证共享变量更新的一致性和可靠性，一个线程应当通过获取一个锁来保证对该变量进行独占的使用，通常这个锁为这些共享变量施加互斥性。

Java语言还提供了第二种机制，即volatile域，对于一些目的，它比锁更加方便。

一个域可以定义为volatile，这种情况下，对于该变量，Java内存模型会保证所有线程会观察到一致的值。

将一个final变量声明同时也定义为volatile会产生编译时错误。

示例，volatile域：

在下面的示例中，一个线程重复地调用方法one（不超过Integer.MAX_VALUE最大次数），另一个线程重复地调用方法two：

class Test {
    static int i = 0, j = 0;
    static void one() { i++; j++; }
    static void two() {
        System.out.println("i=" + i + " j=" + j);
    }
}

然后方法two偶尔会打印出j的值比i的值要大，因为上面的示例中没有同步，那么根据内存模型所描述的规则，共享变量i和j可能不按顺序进行更新。

一种避免这种无序的行为就是将方法one和two声明为synchronized的：

class Test {
    static int i = 0, j = 0;
    static synchronized void one() { i++; j++; }
    static synchronized void two() {
        System.out.println("i=" + i + " j=" + j);
    }
}

这可以避免方法one和two并行地执行，进一步地保证了共享变量i和j在方法one返回前都得到更新。因此方法two永远不会观察到j的值比i的值要大，事实上，它总会观察到相同的值。

另外一种方式是将i和j声明为volatile的：

class Test {
    static volatile int i = 0, j = 0;
    static void one() { i++; j++; }
    static void two() {
        System.out.println("i=" + i + " j=" + j);
    }
}

这使得方法one和two可以并行执行，并且保证对共享变量i和j的访问发生的次数和顺序恰好相同，就像它们出现在被每个线程所执行的程序文本中的那样。因此，共享变量j的值永远不会比i大，因为对i的每次更新必须在对j的更新发生前反映到共享变量i上。然而有可能，对方法two的给定调用会观察到j的值比观察到的i的值要大，因为，在方法two获取i的值的时刻和获取j的时刻之间，方法one可能已经执行了多次。

volatile与原子访问

Java语言底层提供的原子性的操作有如下两种：

  • 对引用变量和大多数基本变量（除了long和double）的读和写都是原子的
  • 对所有声明为volatile的变量（包括long和double）的读和写都是原子的

原子操作不能被交错，因此使用时不用担心线程干扰。但这并不能消除同步原子操作的需要，因为内存一致性错误依然可能存在。使用volatile变量可以降低内存一致性错误，因为任何对volatile变量的写操作都与随后对同一变量的读操作建立了前置发生关系。这意味着对volatile变量的改变对其它线程总是可见的。更进一步，这也意味着，当一个线程在读一个volatile变量时，它不仅可以看到对volatile的最新的改动，也可以看到引起改变的代码的副作用（side effects）。

因此原子性并不能保证内存一致性（可见性）。

对于volatile变量而言，如果一个操作仅涉及对该变量的读与写，不依赖于其它复合性操作，那么它可以保证该变量访问的有序性、可见性和原子性。
