Java内存模型

导论

现代计算机科学中有两个关键的目的，一是让程序运行的更快，在高层级上，这需要数据结构和算法设计。在低层级上，这需要对指令进行重排序以便以最高效的方式运行。二是保证程序员与电脑理解我们让他们做的事。

我们需要一种方法来解决多线程下重排序可能带来的问题，这种方法即我们常说的内存模型，Java内存模型描述程序如何与内存进行交互。程序员所使用的语言的内存模型定义了一直到机器码可能出现的重排序。对于Java这样的高级别的解释性语言，它定义了将代码转换为字节码时编译器可以进行的重排序，将字节码转换为机器码时虚拟机可以进行的重排序，以及处理器执行机器码可以进行的重排序。

我们需要处理因果循环（cause loop）的概念，假设这样的情形： 初始时：x==y==0 Thread 1 | Thread 2 ———|———- r1 = x; | r2 = y; y = r1; | x = r2; 有没有可能 r1==r2==42?

如果线程2中将42写入到x，然后在线程1总读取x，并写入到y，然后线程2读取y，那么42从何而来。

这个例子看上去不切实际，但对一种编程语言的内存模型来说，应当由一个完整的语义来决定是否允许出现这样的行为。这个例子的因果循环便导致了我们不希望看到的行为。

再看一个例子： 初始时：x == y == 0 Thread 1 | Thread 2 ———-|———- r1 = x; | r3 = y; r2 = r1|1;| x = r3; y = r2; | r1 == r2 == r3 == 1 是合理的行为。

这次因果循环导致的行为则是可以接受的，为了看到r1 == r2 == r3 == 1的结果，线程1在读取x前将1写入y，这里似乎线程1在读x之后才能确定r2的值，事实上，编译器可能进行线程间的分析，发现只有0和1会写入到x，那么编译器便可以确定线程1总是将1写入到y。

那么，显然有时因果循环是可以接受的，这类例子的根本问题在于我们不能给出“第一个发生的原因”的理由，关于原因和结果，我们应当考虑得更多。

1.4 方法

一方面，模型有必要允许编程人员精细并正确地解释他们的多线程代码。另一方面，模型有必要让编译器的编写者，虚拟机的设计者和硬件架构人员可以简单粗暴地进行代码优化，这会使得对多线程代码结果的预测不那么直接。也有必要与已有的代码实践和JVM实现相兼容。

我们主要围绕两个目标：

  • 我们需要认真地定义，编程人员如何保证一个线程的动作以一种合理的方式被其它线程可见。
  • 我们需要认真地定义，如果编程人员没有仔细地确保线程以规定的方式进行交互时会发生什么。

第一颗子弹（可以认为是两个目标中更简单的那个）需要对什么构成了同步有一个详细并完整的定义，它允许线程间通讯。锁是两个线程通讯的最显而易见的方式。然而，其它机制也是必要的，例如有时我们需要线程间进行非互斥（mutual exclusion）的协作，从而减少开销，对此，我们引入了volatile域，它将更加传统的内存屏障替换为一个基本的内存一致性操作。最后，我们说明了一种可以忽略数据竞争并保证对象不可变的方法，即final域的使用。

第二颗子弹稍微复杂些，基本问题在上面已经进行了描述：什么样的因果循环是可以接受的，什么是不被允许的。
