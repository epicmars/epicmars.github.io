JVM：class文件校验与Stack Map Frame

除了编译时的检查，为了保证class文件符合规范，JVM实现会在链接时对class文件进行校验。

如果class文件的版本号为49或以下会使用类型推断（Type Inference）进行校验，如果为50.0或以上则会使用类型检查（Type Checking）进行校验。其中版本号50.0是Java 6的版本号，如果版本号恰好为50.0并且类型检查失败了，那么JVM实现可能会尝试采用类型推断进行校验。

从类型推断到类型检查，可以发现类型校验的强度提升了。两种校验方式的区别主要是在class文件的方法信息的代码属性中引入了栈映射帧（stack map frames）信息。其主要目的是为了加速类型校验的过程。

通过类型推断进行校验（Verification by Type Inference）

在链接时，校验器通过对每个方法进行数据流分析堆class文件的每个方法的Code属性的code数组进行检查。它确保在程序的任意给定位置，无论是采用何种代码路径到达该位置，以下条件都为true：

  • 操作数栈总是具有相同的大小并包含相同类型的值
  • 除非知道局部变量数组的一个槽中包含值有一个合适的类型，不然不会对其进行访问
  • 方法是使用合适的类型进行调用的
  • 域采用合适的类型进行赋值
  • 所有操作码在操作数栈和局部变量数组中具有合适类型的参数

通过类型检查进行校验（Verification by Type Checking）

类型检查通过Prolog子句来施加类型规则，它提供了类型规则的一个正式的规范。

对于每个含有一个Code属性的方法，类型检查器需要一个栈映射帧列表。栈映射帧列表由Code属性的StackMapTable属性给出。其意图是一个栈映射帧必须出现在一个方法中的每个基本块的开头。在每个基本块的开头栈映射帧指定了每个操作数栈条目和每个局部变量的验证类型。类型检查器为每个方法读取这些栈映射帧，并使用这些映射来为Code属性中指令的类型安全提供证明。

所谓基本块，即拥有一个初始栈映射帧的块，初始栈映射栈分别存在于方法、跳转指令标签标识的块以及异常处理器块的开始处。

栈映射帧与类型检查

Stack map frame中的信息用于class文件校验过程中的类型检查，ASM这样的字节码操作库可以改变一个方法中的字节码，但仅仅是作字节码上的改变并不能通过类型检查。还需要对栈映射帧信息进行更新。

JVM规范

根据JVM规范中对class文件的描述，栈映射帧（stack map frame）是StackMapTable属性数据结构中的组成部分，而StackMapTable是Code属性的一部分，Code又是class文件中method_info数据结构的一部分。Code，StackMapTable都是属性信息attribute_info，其中attribute_info的通用格式为：

attribute_info {
    u2 attribute_name_index;
    u4 attribute_length;
    u1 info[attribute_length];
}

method_info

method_info {
    u2             access_flags;
    u2             name_index;
    u2             descriptor_index;
    u2             attributes_count;
    attribute_info attributes[attributes_count];
}

其中的属性信息就包括Code，Exceptions，RuntimeVisibleParameterAnnotations,RuntimeInvisibleParameterAnnotations，AnnotationDefault，MethodParameters属性。

Code属性

Code属性属于method_info结构attributes表的元素。一个Code属性包含一个方法中的JVM指令和额外的信息，包括接口初始化方法或一个类的一个实例初始化方法。

Code_attribute {
    u2 attribute_name_index;
    u4 attribute_length;
    u2 max_stack;
    u2 max_locals;
    u4 code_length;
    u1 code[code_length];
    u2 exception_table_length;
    {   u2 start_pc;
        u2 end_pc;
        u2 handler_pc;
        u2 catch_type;
    } exception_table[exception_table_length];
    u2 attributes_count;
    attribute_info attributes[attributes_count];
}

其中的若干数据项的定义如下

  • max_stack

    在方法执行中任意一个时刻其操作数栈的最大深度

  • max_locals

    给出在调用该方法时给局部变量数组分配的变量数，包括传递给方法的参数。 对于long和double类型，其最大的局部变量索引为max_locals - 2，其它类型最大为max_locals - 1

Code属性中包含的属性信息有StackMapTable，LineNumberTable，LocalVariableTable，LocalVariableTypeTable。

StackMapTable属性

StackMapTable属性属于Code属性的attributes表。它用于class文件的验证过程中的类型检查中。一个Code属性的attributes表中最多只可能有一个StackMapTable属性。

StackMapTable_attribute {
    u2              attribute_name_index;
    u4              attribute_length;
    u2              number_of_entries;
    stack_map_frame entries[number_of_entries];
}

该结构的数据项的定义如下：

  • entries

    该表中的每个entry表示方法的一个栈映射帧（stack map frame）。表中栈映射帧的顺序是有意义的。

一个栈映射帧指定了它所应用处的字节码偏移量，以及偏移量对应的局部变量和操作数栈条目的验证类型。

每个栈映射帧依赖于它前面的帧，方法的第一个栈映射帧是隐含的，由类型检查器根据方法描述符进行计算，因此stack_map_frame结构的entries[0]描述了方法的第二个栈映射帧。

一个栈映射帧对应的字节码偏移采用帧中指定的偏移增量offset_delta，并且通过将offset_delta + 1加到前一个帧的字节码偏移量中计算出。除非前一个帧是初始化帧，这种情况下，该栈映射帧对应的字节码偏移就是帧中指定的offset_delta。

通过使用该offset_delta而不是存储实际的字节码偏移，根据定义，我们可以确保栈映射帧以正确的顺序进行排序。

如果一个指令从Code属性的code数组偏移i处开始，并且Code属性有一个StackMapTable属性，其entries数组包含一个对应字节码偏移i的栈映射帧，我们就说字节码中的一个指令具有一个对应的栈映射帧。

一个栈映射帧表示为一个可辨联合stack_map_frame，它包含一个1字节tag，表明它使用联合中那个数据项，后面跟随一个或多个字节，这些字节给出了tag的更多信息。

union stack_map_frame {
    same_frame;
    same_locals_1_stack_item_frame;
    same_locals_1_stack_item_frame_extended;
    chop_frame;
    same_frame_extended;
    append_frame;
    full_frame;
}

它使用如下可辨联合来表示帧中一个局部变量或栈操作数条目的验证类型：

union verification_type_info {
    Top_variable_info;
    Integer_variable_info;
    Float_variable_info;
    Long_variable_info;
    Double_variable_info;
    Null_variable_info;
    UninitializedThis_variable_info;
    Object_variable_info;
    Uninitialized_variable_info;
}
