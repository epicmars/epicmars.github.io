新闻分页问题

问题描述

一个HTTP客户端查看服务端的新闻，新闻根据时间降序排列，客户端按照第1页，第2页，直到第n页的顺序依次加载每个页面。假设客户端已经加载了k（k >= 1）页的新闻，加载过程中服务端的新闻可能会更新，继续加载后续的页面，那么在这一过程中如何保证有效地加载服务器中所有后续页面的内容？

这里需要处理的主要问题是，在用户请求当前页面和下一个页面之间，服务端的新闻可能已经更新了，如果直接根据页面数来做出响应那么会返回已请求过的数据，如下图所示，当客户端请求第k页的数据时，服务端可能返回之前页面的数据。

请求第k页：
client request:  |--------page(k)----------|---------page(k+1)------|
server response: |--------page(k)----------|---------page(k+1)------|

请求第k+1页：
client request:  |--------page(k)----------|---------page(k+1)------|
server response: |-----new data--|--...--|--------page(k)----------|---------page(k+1)------|

服务端处理

API设计

不根据页面来进行分页，因此，需要额外的信息来处理下一次响应，目前作如下几种考虑：

  • 根据服务端新闻id来处理
    1. 服务端还可以记录用户请求第一个页面时第一条新闻的id，这样可以在排序中计算出已更新的新闻数量，那么查询下一页时只需在原来页面offset基础上加上这个数量就可以。
    2. 服务端可以不按页面信息来给出响应，而是记录用户上一次请求的最后一个新闻id，然后请求该id之后的一个页面。

对于资讯流，如微博，twitter等，这种分页问题已有一些方案，可以参考twitter Api。

添加缓存

为提高请求性能，可以利用Redis等类似中间件搭建服务端的缓存，并为缓存设计过期时间，定时进行缓存更新。

客户端处理

在移动应用开发中，客户端可以设计类似Cookie和Session的机制来保存请求的信息，但这会增加维护的成本，一旦业务发生变化，之前的缓存机制有很大可能会弃用。并且相比固定宽带，移动端流量更加昂贵，尤其是资讯、图片等应用中，一般需要利用缓存机制来节约流量，也就是缓存优先的策略，只有在缓存数据过期了才去请求服务器。那么在缓存优先的情况下如何处理上述的分页问题。

缓存优先策略

仍然主要处理请求过程中服务端新闻已经更新的问题，考虑客户端请求第k页的新闻，假设服务端未设计会话机制，总是返回服务器中的第k页数据，那么第k页数据可能包含已请求的数据，也可能只有最近更新的数据。采用一种Android架构设计中推荐的方法：

客户端的分页这样设计，采用单一数据源的原则，每次请求总是先查询缓存中的数据，根据缓存的数据和请求信息等判断是否需要请求网络。这里的缓存策略如下：

  • 查询缓存后如何判断是否请求网络
    • 如果是第一页，或者缓存中的数据为空
    • 缓存不为空，但缓存失效了
  • 如果发生了网络请求，那么缓存请求的数据，然后再次从缓存中查询当前页面数据

缓存失效机制

那么如何设计缓存失效机制，可以有两种实现机制:

  • 过期机制，添加过期时间，页面中的新闻过期了即重新请求页面并更新页面中新闻的过期时间。
  • 内容更新机制，如果不使用过期机制，如何去判断缓存失效了，目前仅有的信息是请求的页面数，其它数据字段，包括服务端数据id，用于排序的新闻发布时间都是不可靠的。那么既然没有过期时间，现在需要判断当前的页面已经失效了，也就是与服务端不同步了，此时服务端的页面与客户端缓存的页面是不同的。因此可以将页面信息保存在缓存中，一旦客户端与服务端页面信息不匹配表明当前页面失效，需要重新请求。每次更新后将页面信息也缓存起来。

采用内容更新机制的实时性比过期机制更好，一旦服务端数据更新了客户端可以马上做出反应。

扩展与优化

按照上面的内容更新机制来作缓存和分页，如果服务器频繁地进行更新，那么每次即使仅更新一条新闻就会导致所有页面变动，导致后续每个分页的缓存失效。因此需要更加精确的内容更新机制，由于新闻是严格根据时间按由新到旧的顺序进行排序的，当下载一个最新的页面后，如果最新下载的页面有内容已经缓存过了，说明最新页面与已缓存的页面是有重合的，那么下一页的缓存可能仍然有效。

那么问题便成了什么情况下下一页缓存仍然有效？一种判断是在将有重复数据的页面保存到缓存中时会覆盖缓存中当前页面部分或全部数据，这里是一个临界页面，抛开页面来分析如果缓存中的数据是连续的那么就是有效的，因此对于下一页缓存中的数据是否是连续的，需要判断它与上一页是否连续，以及它自身的数据是否连续。

因此现在问题转换为一个缓存页面数据是否是连续的问题？由于缓存中的页面根据缓存时间段的不同，由若干连续的段组成，这些不同的段中的页面是连续的，由于新闻严格按照时间排序，这些段内部的数据也是连续的，但不同的段之间可能不连续。

下面是用户视角的分页与数据库已有缓存分页的关系：

用户 page i page_size_user

page_size_user <= page_size_database
-------i------|----------i+1--------|

page_size_user == page_size_database
------------i-----------|-----------i+1----------|

page_size_user > page_size_database
-------------------i-----------|---------------i+1-------------------|

数据库 page x y z page_size_database
------------x-----------|------------y-----------|-----------z----------|

因此，对页面缓存与上一页是否连续，以及其自身是否连续的分析中，在判断用户视角所对应的第i+1页的缓存数据是否连续时，需要用户视角对应的i页的缓存数据最后一条新闻的页，以及第i+1页缓存中所有数据的页面，然后判断这些页面是否连续。为了方便将第i页缓存最后一条数据合并到i+1页缓存数据，作为一个整体来进行分析其连续性，如果第i页没有对应的缓存，也就是没有覆盖发生，认为最后一条数据的页面标记为null，它与任何页面包括其它标记为null的页面都不连续。

如果业务有变动，需要设计一个周全的方案，例如页面的大小可能变动，或者段的部分页面被新的数据覆盖导致段变短，因此已更新的页面或者下一个缓存页面可能会包含不同时间段的页面，但一个段内的页面总是相等或递增的。为了确保分析的完备与正确，只需将这些不同的段看做更大的页面集合。继续分析缓存页中数据是否连续的问题，考虑几种情况：

  • 缓存数据的页面均为0，这种情况可能是不同段均只有第一页的数据，可能不连续。认为缓存不连续，需要重新下载。
  • 缓存数据的页面均为k（k>0），连续
  • 缓存数据的页面不全相同但是递增的
    • 第一页是0
      • 有多条数据页面为0，与均为0的情况类似，可能不连续
      • 仅第一条数据页面为0，连续
    • 第一页是k(k>0)，如果连续页面位于同一段中，那么连续成立。如果连续页面位于不同段中，那么这种连续必定是由于页面重合导致的，连续也成立。
  • 缓存数据页面不全相同也不是递增的，页面必然位于不同段中，这种情况下仅在相邻段之间均有重合的情况下才是连续的
    • 第一条数据为0，其余页面均大于0
    • 所有页面均大于0

通过分析可以得出，由于一个段的页面是递增的，最初一个段的起始页面数为0，新的段与其发生了重合，那么其必然有部分页面数为0的数据被覆盖了。但现在仅根据被覆盖的最后一条数据来判断两个段是否连续，需要排除前一个段的最后一个页面数也为0的情况。因此判断连续的条件可总结为：第一条数据页面大于等于0，其余数据页面均大于0。由于页面数总是非负数，因此判断连续的条件即为缓存数据页面均大于0。

非连续页面处理

对于缓存中的一个连续的页面，直接返回缓存中的数据即可。如果是一个非连续的缓存页面，会先请求网络并将这部分请求数据保存到缓存中，然后再从缓存中获取数据。请求的数据总是在当前缓存页面之前的，分为如下几种情况：

  • 从网络请求到的数据都没有被缓存过，这样会在当前缓存页面之前插入一个新的页面
    • 新插入的页面位于上一个页面之后，也就是与上一个页面连续，那么再次获取第k页会是最新的数据，不需要特别的处理。
    • 新插入的页面位于第一页之前，那么在保存请求数据后，再次获取第k页的缓存时会是上一次获取的第k-1页的缓存。
  • 从网络请求到的数据全部都是已缓存过的，也就是对当前获取的缓存页面没有影响，那么重新获取的缓存页面保持不变，仍然是不连续的。
  • 从网络请求到的数据有部分已经缓存过了，部分是新的数据
    • 新的数据位于页面后面，那么它也与上一个页面连续。那么重新获取的第k页的缓存中，这部分新的数据是连续的，后面的数据仍有可能不连续。
    • 新的数据位于页面前面，那么新的数据在缓存第一页之前。那么重新获取的第k页缓存数据包含前一页缓存的与新插入数据数量相等的部分数据，并且剩余的部分数据可能是连续的，也可能部分连续，也可能从开始就不连续。

那么说明即使缓存页被判断为非连续的，也就是失效了，并且从服务器请求了新的数据，由于服务器与缓存中的数据可能不一致。在将请求缓存后重新获取缓存页，该缓存页可能会存在如下几种异常情况： - 是一个旧的页面 - 仍然是之前用户判断为非连续的那个缓存页面页面 - 含有部分与上一页连续的数据，剩余的部分可能不连续 - 含有部分与上一页重复的数据，剩余的部分可能不连续

总结

到此，将所有这几种情况考虑在内，可以发现一旦上述情况发生，不能保证最终返回的缓存页与上一页是连续的。即使，根据对上述异常页面分别处理，如在得到旧的数据时进行忽略，返回空页面（保持页面不变）或仅返回部分连续的数据（增加偏移量并保持页面数不变）。这明显增加了维护的复杂度，并易于出错。因此最终客户端分页方案仅保留缓存失效机制，分页机制仍然以上述服务端为准。
